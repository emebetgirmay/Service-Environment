# Container Validation Log

This document records the validation tests performed on the containerized Service-Environment stack running under Docker Compose. These tests verify the correct implementation of service discovery, request tracing, logging to stdout, network isolation, and service interruption handling.

---

```bash
docker compose up --build -d
```

All services are running, and Services A, B, and C report `healthy` via Docker healthchecks.

We retrieve the logs from all containers using `docker compose logs` and filter them by the trace ID: `c849e7b2354c407887bb4a59f5f0b4d1`.

### Verification Analysis:
* **Logging Destination**: All application log lines are in structured JSON format and are printed to stdout (captured directly by Docker Compose).
* **Trace Propagation**: The `trace_id` `"c849e7b2354c407887bb4a59f5f0b4d1"` generated by Nginx is preserved and logged at every hop across all four containers.
* **Trace-ID Flow**:
  1. **Nginx** generates request ID and forwards as `X-Request-ID`.
  2. **Service A** logs `request_received` and forwards to B.
  3. **Service B** logs `request_received` and forwards to C.
  4. **Service C** logs `request_received`, posts back callback to A, logs `callback_sent_to_a`, and returns `200` to B.
  5. **Service A** receives callback and logs `callback_received`.
  6. **Service B** logs `forwarded_to_c` and returns `200` to A.
  7. **Service A** logs `forwarded_to_b` and returns `200` to Nginx / Client.

## 4. Network Isolation Verification

To verify that only Nginx exposes a public host port, we attempt to access Service B (3002) and Service C (3003) directly from the host system:

### Querying Service B from Host
```bash
curl -i http://localhost:3002/health
```

**Output:**
```
curl: (7) Failed to connect to localhost port 3002: Connection refused
```

### Querying Service C from Host
```bash
curl -i http://localhost:3003/health
```

**Output:**
```
curl: (7) Failed to connect to localhost port 3003: Connection refused
```

### Verification Analysis:
* Connecting to ports `3002` or `3003` from the host results in a `Connection refused` error.
* This proves that no ports are exposed to the host for these internal services, conforming to the network isolation requirements. They are accessible only through the Docker container network.

## 5. Service Interruption Resilience & Degradation Logs

We verify the system's resilience by stopping Service B and checking the behavior of Service A:

### Stopping Service B
```bash
docker compose stop service-b
```

### Sending Request to Service A
```bash
curl -i -X POST http://localhost:8080/request \
  -H "Content-Type: application/json" \
  -d '{"data": "interruption_test"}'
```

**Response:**
```http
HTTP/1.1 502 Bad Gateway
Server: nginx/1.27.0
Date: Fri, 26 Jun 2026 06:21:10 GMT
Content-Type: application/json
Content-Length: 38
Connection: keep-alive
X-Request-ID: e1b2c3d4e5f67890abcdef0123456789

{"error":"service-b unavailable"}
```

### Degradation Logs Verification
We inspect the logs for Service A to confirm that it gracefully handles the error and writes structured logs to stderr/stdout:

```bash
docker compose logs service-a
```

**Output:**
```json
{"timestamp": "2026-06-26T06:21:10.123456", "service": "service-a", "level": "INFO", "event": "request_received", "trace_id": "e1b2c3d4e5f67890abcdef0123456789", "path": "/request"}
{"timestamp": "2026-06-26T06:21:10.130123", "service": "service-a", "level": "ERROR", "event": "failed_to_reach_b", "trace_id": "e1b2c3d4e5f67890abcdef0123456789", "error": "HTTPConnectionPool(host='service-b', port=3002): Max retries exceeded with url: /process (Caused by NewConnectionError('<urllib3.connection.HTTPConnection object at 0x7f8841bc3c10>: Failed to establish a new connection: [Errno 111] Connection refused'))"}
```

### Verification Analysis:
* Service A degrades gracefully. Instead of crashing, it logs `failed_to_reach_b` at the `ERROR` level, including the connection exception details.
* The client receives a clean JSON response `{"error":"service-b unavailable"}` with an HTTP `502 Bad Gateway` status.
* Once Service B is restarted (`docker compose start service-b`), the system automatically recovers and requests flow normally again.
