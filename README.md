# Service-Environment

## Project Overview

This is a small production-style internal service environment consisting of three independent HTTP services (Flask) and an Nginx reverse proxy:

- **Service A** (port 3001) — the only publicly reachable service. Accepts incoming client requests, initiates a call to Service B, and receives a callback from Service C once processing completes.
- **Service B** (port 3002) — internal-only. Receives requests from Service A and forwards them to Service C.
- **Service C** (port 3003) — internal-only. Receives requests from Service B and calls back to Service A to report completion.
- **Nginx** — the single public entry point on port 80, reverse-proxying all traffic to Service A only.

Every request is traceable end-to-end via a shared `X-Trace-ID` header, and every service emits structured JSON logs.

## Architecture

```
Client
  │
  ▼
Nginx (port 80, public)
  │  proxy_pass → 127.0.0.1:3001
  ▼
Service A (3001) ── POST /request
  │  forwards to service-b.internal:3002
  ▼
Service B (3002) ── POST /process
  │  forwards to service-c.internal:3003
  ▼
Service C (3003) ── POST /execute
  │  calls back to service-a.internal:3001
  ▼
Service A (3001) ── POST /callback   (completes the loop)
```

- All three services run on the same host and bind to `127.0.0.1` only — they are not reachable from outside the machine at the socket level (see [Network Security](#network-security)).
- Services address each other by hostname (`service-b.internal`, `service-c.internal`), resolved locally rather than hardcoded by IP (see [Service Discovery](#service-discovery)).
- A single `X-Trace-ID` header is generated by Service A (or passed through if already present) and forwarded on every downstream call, so one client request can be followed through all four hops in the logs (see [Request Tracing](#request-tracing)).

## Installation

Current status: there is **no automated install script yet** — deployment is manual. Steps to set up on a fresh Ubuntu machine:

1. Clone the repo:
   ```bash
   git clone https://github.com/emebetgirmay/Service-Environment.git
   cd Service-Environment
   ```
2. Create a virtual environment and install dependencies (no `requirements.txt` is committed yet — install directly):
   ```bash
   python3 -m venv venv
   venv/bin/pip install flask requests
   ```
3. Add local service-discovery hostnames to `/etc/hosts`:
   ```bash
   echo "127.0.0.1 service-a.internal
   127.0.0.1 service-b.internal
   127.0.0.1 service-c.internal" | sudo tee -a /etc/hosts
   ```
4. Install and configure Nginx:
   ```bash
   sudo apt update && sudo apt install -y nginx
   sudo cp nginx/service-environment.conf /etc/nginx/sites-available/service-environment.conf
   sudo ln -s /etc/nginx/sites-available/service-environment.conf /etc/nginx/sites-enabled/
   sudo rm -f /etc/nginx/sites-enabled/default
   sudo nginx -t && sudo systemctl reload nginx
   ```
5. Configure the firewall (see [Network Security](#network-security) for why):
   ```bash
   sudo ufw allow OpenSSH
   sudo ufw allow 80/tcp
   sudo ufw default deny incoming
   sudo ufw default allow outgoing
   sudo ufw --force enable
   ```

## Operation

**Current status:** services are run as plain foreground Python processes, not yet under systemd. There is no automatic start-on-boot or restart-on-failure — this is the main known gap in the current implementation.

**Start services** (each in its own terminal, from the repo root):
```bash
venv/bin/python service-a/app.py
venv/bin/python service-b/app.py
venv/bin/python service-c/app.py
```
Order matters operationally even though it isn't enforced automatically yet: start B and C before A, since A's `/request` flow depends on B being reachable.

**Stop services:** `Ctrl+C` in each terminal (or `pkill -f "service-a/app.py"` etc.)

**Restart services:** stop, then start again as above.

**Verify health:**
```bash
curl http://127.0.0.1:3001/health
curl http://127.0.0.1:3002/health
curl http://127.0.0.1:3003/health
```
Each should return `{"service": "...", "status": "ok"}`.

## Validation

To confirm the full system is working end-to-end:

1. Confirm all three health endpoints return `200` (see above).
2. Send a request through Nginx (the intended public entry point):
   ```bash
   curl -X POST http://127.0.0.1/request -H "Content-Type: application/json" -d '{"hello":"world"}'
   ```
3. Watch the terminal output of all three services — you should see, in order:
   - Service A logs `request_received`, then `forwarded_to_b`
   - Service B logs `request_received`, then `forwarded_to_c`
   - Service C logs `request_received`, then `callback_sent_to_a`
   - Service A logs `callback_received` — confirming the full A→B→C→A loop completed
4. All four log lines should share the same `trace_id`, confirming the request was traced across every hop.
5. Confirm internal services are not externally reachable (see [Network Security](#network-security)).

## Logging

All three services log structured JSON to **stdout** (visible directly in the terminal each service is running in). Each log line is a single JSON object with:

- `timestamp` — ISO 8601 UTC
- `service` — which service emitted it (`service-a`, `service-b`, `service-c`)
- `level` — `INFO`, `WARN`, or `ERROR`
- `event` — what happened (e.g. `request_received`, `forwarded_to_b`, `invalid_route`, `failed_to_reach_b`)
- `trace_id` — the request's trace ID, consistent across all services for a given request
- additional context fields depending on the event (e.g. `status`, `error`, `path`)

This answers the standard production-debugging questions: what happened, when, which service, which request, and what was the outcome.

Nginx access/error logs are written to `/var/log/nginx/service-environment.access.log` and `/var/log/nginx/service-environment.error.log` (per the `access_log`/`error_log` directives in `nginx/service-environment.conf`).

Since services aren't under systemd yet, `journalctl` isn't currently a log source — that will change once systemd units are added.

## Request Tracing

Every request carries an `X-Trace-ID` header:

- Nginx sets `X-Trace-ID: $request_id` on the way in, giving Nginx's own internally-generated unique ID to the first hop.
- Service A reads `X-Trace-ID` from the incoming request (falling back to a freshly generated UUID if absent) and includes it in every log line and every downstream call it makes.
- Service B and Service C do the same: read the incoming `X-Trace-ID`, log it, and forward it on their own outbound calls.
- Service C's callback to Service A also carries the same `X-Trace-ID`, so the final hop in the loop is tied back to the original request.

To trace a single request: pick its `trace_id` from any one log line, then `grep` for it across all terminals/log files to see every step the request took.

## Service Discovery

Services communicate using logical hostnames instead of hardcoded IP addresses:

- `http://service-a.internal:3001`
- `http://service-b.internal:3002`
- `http://service-c.internal:3003`

**How it works:** these hostnames are resolved via static entries in `/etc/hosts`, each pointing to `127.0.0.1` (all services run on the same host in this environment):

```
127.0.0.1 service-a.internal
127.0.0.1 service-b.internal
127.0.0.1 service-c.internal
```

**What performs the resolution:** glibc's name service switch (NSS). When a service calls `requests.post("http://service-b.internal:3002/...")`, the C library's `getaddrinfo()` call resolves the hostname according to `/etc/nsswitch.conf` (`hosts: files dns`), which checks `/etc/hosts` before ever querying DNS. Since the name is found in `/etc/hosts`, resolution never leaves the local machine.

This is a static, host-file-based discovery mechanism — appropriate for this single-host environment. In a multi-host or container deployment, this would be replaced by real DNS-based discovery (e.g., a DNS server, Docker's embedded DNS, or a service mesh like Consul), but the calling code wouldn't need to change since it already resolves by name, not by IP.

**Troubleshooting service discovery / name resolution failures:**

1. Confirm the hostname resolves at all: `getent hosts service-b.internal`
2. If it doesn't resolve, check the entry exists: `cat /etc/hosts | grep internal`
3. Check resolver order: `cat /etc/nsswitch.conf | grep hosts` — should show `files` before `dns`
4. Test the resolved address actually answers: `curl -v http://service-b.internal:3002/health`
5. If resolution works but the request still fails, the problem is downstream (process not running, port not listening, firewall) — not discovery. Use `ss -tlnp | grep 300` to confirm the target service is actually listening.

## Network Security

Service A is the only service meant to be publicly reachable (via Nginx on port 80). Service B (3002) and Service C (3003) are internal-only and must not be reachable from outside the VM.

**Why they're protected:** B and C only exist to be called by other internal services in the request chain (A → B → C → A). They have no reason to accept traffic from the public internet, and exposing them would let an attacker skip Service A entirely and hit internal processing logic directly.

**What enforces the protection (defense in depth, two layers):**

1. **Socket binding:** each Flask service binds explicitly to `127.0.0.1` (see `app.run(host="127.0.0.1", ...)` in each `app.py`). This means the OS never opens a listening socket on any external network interface for ports 3001–3003 — they are unreachable from outside the machine regardless of firewall state.
2. **Firewall (UFW):** as a second, explicit layer, UFW is configured with a default-deny-incoming policy, allowing only:
   - `22/tcp` (SSH, for management)
   - `80/tcp` (Nginx, the only public entry point)

   Ports 3001–3003 are not explicitly allowed, so they fall under the default deny rule for any external interface. Loopback traffic (`127.0.0.1`) is unaffected by UFW, so Nginx → Service A and the internal A→B→C→A chain continue to work normally.

**How to verify the protection:**

- From the VM itself: `curl http://127.0.0.1:3002/health` → succeeds (loopback, by design)
- From outside the VM (e.g., another machine on the network or via the VM's public/LAN IP): `curl http://<vm-ip>:3002/health` → should hang/time out or refuse — confirms B/C are not externally reachable
- Check firewall state: `sudo ufw status verbose` — should show only 22 and 80 allowed, default deny incoming
- Confirm Service A *is* reachable externally through Nginx: `curl http://<vm-ip>:80/health`

**Troubleshooting connectivity issues:**

1. If the instructor can reach B/C externally: check `sudo ufw status` — confirm no broader-than-intended rule (e.g., `3000:3010/tcp ALLOW`) was added by mistake.
2. If internal service-to-service calls start failing after a UFW change: remember UFW does not filter loopback (`lo`) traffic by default — if internal calls break, the cause is more likely the Flask process not running or a hostname resolution issue, not the firewall. Check with `ss -tlnp | grep 300` to confirm the process is bound and listening.
3. If Nginx can't reach Service A: confirm Service A is listening on `127.0.0.1:3001` (`ss -tlnp | grep 3001`), and that Nginx's `proxy_pass` target matches.
4. General network triage order: is the process running? → is it listening on the expected interface/port? → does the hostname resolve? → does the firewall allow it? → does the proxy config point to the right place?

## Troubleshooting

Beyond service discovery and network access (covered above), here's how to investigate other classes of failure:

**Service startup failures**
- Run the service directly in the foreground (`venv/bin/python service-a/app.py`) and read the traceback — Flask will print import errors, port-already-in-use errors, etc. immediately.
- Check if the port is already taken: `ss -tlnp | grep 3001`.

**Service dependency failures** (e.g. Service B or C is down)
- Service A's `/request` handler wraps its call to B in a try/except; if B is unreachable, A logs `failed_to_reach_b` with the exception message and returns HTTP `502` to the client. Same pattern for B→C (`failed_to_reach_c`) and C→A (`failed_to_callback_a`).
- To diagnose: grep the failing service's logs for `ERROR` level entries, check the `error` field for the underlying exception (connection refused vs. timeout vs. DNS failure point to different root causes), then confirm the dependency process is actually running (`ss -tlnp | grep 300<n>`) and healthy (`curl .../health`).
- Currently there is no automatic dependency ordering on startup (no systemd `After=`/`Wants=` yet) — if services are started in the wrong order, A may simply fail requests until B/C come up; this is a known gap, not expected production behavior.

**Reverse proxy failures**
- `sudo nginx -t` validates the config syntax before reloading.
- Check Nginx's own error log: `sudo tail -f /var/log/nginx/service-environment.error.log` — connection-refused errors here usually mean Service A isn't running or isn't listening on `127.0.0.1:3001`.
- Confirm only one `proxy_pass` exists per `location` block — multiple `proxy_pass` directives in the same block are invalid; only the last one silently takes effect.

**Missing logs**
- All three services log to stdout — if you don't see log lines, confirm the service is actually running in that terminal (not a stale/different process) and that the request actually reached it (check Nginx access log first to confirm the request left Nginx).
- Logging uses `print(..., flush=True)`, so output should appear immediately, not buffered.

**Invalid routing behavior**
- Each service has a 404 handler that logs a `WARN`-level `invalid_route` event with the requested `path` and `trace_id`, and returns a JSON `404` body. If you hit an unexpected route and see no log line, the request likely didn't reach the Flask app at all (check Nginx/firewall first).

**Inter-service communication failures**
- Use the `trace_id` from the failing request to correlate logs across all three services and pinpoint exactly which hop failed and why (timeout, connection refused, unexpected status code).
- `curl -v` directly against the suspected failing hop's hostname/port (e.g. `curl -v http://service-c.internal:3003/execute -d '{}'`) to isolate whether the problem is network-level or application-level.

**Known current gaps** (being addressed separately, noted here for transparency):
- No systemd unit files yet — services don't start on boot or restart automatically on failure, and dependency ordering (A after B/C) is not enforced by the OS.
- No `requirements.txt` committed.
- `venv/` is currently tracked in git, which is not best practice and has already caused one pull conflict.
